# CardWeaver アーカイブ・開発記録 (Archive / Dev Log)

このファイルは、システムのアーキテクチャ設計意図および過去の試行錯誤（開発履歴）を一元管理するためのアーカイブです。
今後のアップデートや動作変更の履歴もすべてここに時系列降順（新しいものが上）で追記していきます。

## 開発履歴 (Trial & Error History)

### [2026/04/17 引き続き] OSS拡張性のためのディレクトリ整理と大規模ドキュメンテーション追加
- **課題:** 「ブラックボックスを排除しアイデアをローカルで管理したい」という目的に賛同した他の開発者がプラグイン（独自機能）を追加したいと思った際、ルールや構造が分からず参入障壁が高かった。また、ルートディレクトリにまだ `FormMain.cs` などのデザインファイルやリソースが散乱していた。
- **対処・結果:** 
  - メインフォーム群（`FormMain.cs`, `.Designer.cs`, `.resx`, `.Events.cs`）をすべて `Forms/` ディレクトリに移動し、名前空間を `CardWeaver.Forms` へ修正してディレクトリ構造を完全にクリーン化した。
  - プログラム全域（特に `CardControl.cs`, `BoxControl.cs`, `WorkspaceManager` など）に大量の C# 標準XMLコメント（`/// <summary>`）を追記した。
  - コメントの内容には「UIを変更した際は `WorkspaceData` をどう更新すればUndoやJSON保存にシームレスに追従できるか」といった OSS開発者向けの手引き を明記した。

### [2026/04/17 深夜] 自動整列機能（Auto-Align）を GridLayoutGroup 数学モデルへ改修
- **課題:** 自動整列を実行すると、ボックスに関係なくキャンバスの左上から一列に並んでしまい、ボックスに収めていた「論理的な空間記憶」もすべて解除されて散らかってしまっていた。
- **対処・結果:** 
  - メソッド `AutoAlignCards` を大幅に書き直し、`parentMap` と連動させた。
  - ボックス内に属するカードについては、親ボックスの現在の「横幅」から逆算した列数（`cols`）を数学的に割り出し、左上から格子状（グリッド）に綺麗にパッキングするようにした。
  - ボックス内にカードが入り切らなくなった（高さがオーバーした）場合は、ボックスの縦幅（`Height`）を自動的に拡張させる処理を導入。
  - どのボックスにも属さない「フリーカード」については、存在しているすべてのボックスを避けるため、空間内の最下部にまとめて整列させる仕様にして見栄えを整えた。

### [2026/04/17 後半] 構造大幅リファクタリング、Undo/Redo・自動整列の追加
- **課題:** UI以外の処理（ファイルの保存、Z-Order管理、イベント処理など）が `FormMain.cs` に密集し、900行近い「神クラス」になりかけていた。また、ボックスの中にボックスが完全に重なると勝手に一緒に動いてしまう過剰なネスト問題や、操作ミスの取り消しができない問題があった。
- **対処・結果:** 
  - `FormMain.cs` を解体し、ファイル保存を `Managers/WorkspaceManager.cs` に、元に戻す機能を `Managers/HistoryManager.cs` に、線描画を `Forms/LineOverlayForm.cs` にそれぞれ分離した。イベント登録ロジックも `FormMain.Events.cs` (partial class) に逃がして見通しを大幅改善。
  - ボックスのネスト追従ロジックを削除。「重なる」ことはできるが支配関係（親子関係）を持たない安全な仕様に変更。
  - Undo/Redo の実装に向けて、変更を検知する `DataChanged` イベントを子コントロールに追加。JSONスナップショットベースで履歴を全保存・復元する堅牢な機構を導入。
  - キャンバス上のカードを O(N \log N) ソートを用いて再配置する仕組みを構築し、「自動整列」ボタンを追加した。
  - Gitリポジトリがキャッシュファイルで膨らんでいたため、`.gitignore` を標準的なクリーンリストに置き換えた。

### [2026/04/17 前半] UI調整、パンニング時のガビガビ解消、コピペの実装
- **課題:** キャンバスをマウスで掴んでドラッグ移動（パンニング）した際、大量のコントロールの再描画が衝突し、画面に激しいチラつき（ティアリング）が発生していた。また、Ctrl+C / Ctrl+V や新規作成機能が欠如していた。
- **対処・結果:** 
  - `FormMain` の `CreateParams` をオーバーライドし、`WS_EX_COMPOSITED` (0x02000000) による高度なダブルバッファリングを導入して移動中のガビガビを完全に解消。バターのような滑らかさを実現した。
  - `FormMain` コントロール全体で `KeyPreview = true` を設定し、最後にクリックされた対象のカードのモデル情報をインメモリでコピー、カーソル近辺にペーストする機能を実装した。
  - UIを見直し、文字が見切れていたボタンの調整と、キャンバスの完全初期化機能（新規作成）を追加した。

### [プロジェクト初期] ベースアプリケーション構築、Z-Order問題の解決
- **課題:** C# WinForms環境でのインフィニットキャンバス制御と、直感的な前面・背面（Z-Order）挙動の実現。
- **対処・結果:**
  - `BringToFront()` を用いた優先順位カスタムロジック（`UpdateZOrder`）を実装。「カードは必ずボックスの上にあり、最近ユーザーが触ったものがさらに上に来る」という直感的な管理を達成。
  - 自由配置可能な `CardControl` と `BoxControl` クラスを独立して定義し、ドラッグ＆ドロップ移動、エッジによるサイズ変更、テキスト編集、右クリックによるカラー変更（ContextMenuStrip）を完備。
  - カード同士を線で繋ぐロジックのベースの描画エンジンを作成。

## アーキテクチャ設計・行動指針 (Architecture Directives)

本プロジェクトでは、コードの品質と継続的な開発を支えるため、以下の6つの原則に基づく設計を採用しています。今後コードを追加する際もこの思想に従います。

**1. 保守性 (Maintainability) 「直しやすさ」**
ファイルを Managers, Forms など役割ごとに分割し、1箇所を修正した際の影響範囲（バグの伝播）を最小化します。クリーンな `.gitignore` 環境を保ちます。

**2. 可読性 (Readability) 「読みやすさ」**
`partial class` や直感的なメソッド名を採用し、コメントが無くても「どんなデータを渡し、どう返すか」の全体の処理フローが読み手に一目で伝わるように構成します。

**3. 可用性 (Availability) 「壊れにくさ」**
誤操作からいつでも復帰できるよう Undo/Redo を実装済みです。メモリ空間の参照崩壊を防ぐため、スナップショット（JSON）によるステートの完全置換を標準手段とします。

**4. 拡張性 (Extensibility) 「足しやすさ」**
`WorkspaceData` という単一のデータモデルに全ての状態を集約しています。新しい要素（画像、付箋、矢印）を追加する場合も、モデルとシリアライザを拡張するだけでセーブ・ロード・Undoにフル対応できる土台となっています。

**5. 信頼性 (Reliability) 「正しさ」**
「重なったから勝手にグループ化される」といったユーザーが予感しえない自動的すぎる挙動を排除し、常に視覚（Z-Order）と操作が1対1に紐づく直感的な動作に留めます。

**6. 効率性 (Efficiency) 「速さ・軽さ」**
ソートや座標再結合のアルゴリズムは軽量・高速なものを選定し、`Invalidate` (再描画) のタイミングを厳密にコントロールすることで無駄なCPU/メモリの消費を防止し、大キャンバスでも重くならないように設定します。
